01. Agent에게 일을 맡긴다는 건

실행 정의가 없으면
AI는 답변은 해도 일을 끝내지 못합니다

AS IS

“이거 잘 정리해줘”

  • 무엇을 판단해야 하는지 모호합니다.
  • 어떤 데이터를 봐야 하는지 흩어집니다.
  • 결과가 나와도 검수 기준이 없습니다.
→
TO BE

“이 자료를 읽고,
이 기준으로 판단해,
이 형식으로 끝내줘”

  • 의사결정 데이터가 정해집니다.
  • 실행 순서가 작업 순서 파일에 남습니다.
  • 검수와 통제가 가능한 결과물이 나옵니다.

02. 잘하는 사람의 방식

Agent 온보딩은 네 가지 문장으로 압축됩니다

역할

어떤 담당자인가

블로그 리뷰단처럼 맡길 일을 좁게 정합니다.

판단 기준

무엇을 잘해야 하나

좋은 결과를 판단할 기준을 문장으로 남깁니다.

금지선

무엇을 하면 안 되나

없는 사실, 과장, 확정하면 안 되는 일을 막습니다.

검수 방식

어떻게 확인하나

사람이 바로 검수할 결과 형식과 질문을 정합니다.

다음 사례이 네 문장을 기준으로 블로그 리뷰단 Agent를 해부합니다. 입력 자료, 반복 기준, 작업 순서, 검수 기준이 어떻게 나뉘는지 관찰합니다.

블로그 리뷰단 Agent의 결과물은
글 하나가 아니라 발행 패키지입니다

기획 묶음

방향을 먼저 정리

타겟 독자, 메시지, 제목 후보, 키워드 방향을 먼저 잡습니다.

채널 묶음

채널별로 다르게 변환

블로그 본문, 인스타 캡션, 첫댓글, 해시태그를 각각의 문법으로 만듭니다.

검수 묶음

발행 전 확인

없는 경험, 과장 표현, 조건 누락, 말투 이탈을 표로 확인합니다.

03. 블로그 리뷰단

단순 QA가 아니라
우리만의 검수 기준을 남기는 Agent입니다

CORE STANDARD 결과를 고치는 AI가 아니라
기준을 남기는 구조 좋은 Agent는 결과물을 한 번 수정하는 데서 끝나지 않고, 다음 결과물에도 적용될 검수 기준을 남깁니다.
사실성방문하지 않은 경험, 확인되지 않은 효과, 과장된 감정을 만들지 않는다.
분리업체 조건과 내 실제 후기를 섞어 사실처럼 쓰지 않는다.
말투기존 블로그의 문장 길이, 표현 강도, 문단 리듬을 유지한다.
채널블로그, 인스타, 첫댓글이 각각 다른 채널 문법을 따르게 한다.
발행성최종본은 사람이 바로 복사해 쓸 수 있는 발행 패키지로 끝낸다.

04. 업무 폴더 설계

작업 순서 파일은 다섯 줄부터 채우면 됩니다

01 목표

끝나는 모습을 씁니다

이 작업 순서가 끝났을 때 무엇이 만들어져야 하는지 한 문장으로 씁니다.

목표: 후기 메모를 발행 패키지로 바꾼다.
02 읽을 파일

자료 위치를 지정합니다

AI가 먼저 읽어야 할 폴더와 파일 이름을 적습니다.

읽을 파일: 00_admin, 01_이번재료, 02_반복기준
03 작업 순서

내 판단 순서를 적습니다

사람이 머릿속으로 하던 순서를 번호로 풀어냅니다.

순서: 읽기 → 분리 → 작성 → 교정
04 금지선

하면 안 되는 일을 씁니다

없는 경험, 과장 표현, 확정하면 안 되는 내용을 막습니다.

금지: 없는 경험 생성 금지
05 출력/검수

확인 가능한 형태로 받습니다

최종 결과물과 검수표를 함께 요청합니다.

출력: 본문 + 캡션 + 검수표
채용 비유로 이해하기처음부터 “채용 업무 전부 해줘”라고 맡기지 않듯이, 작업 순서도 “JD 기준으로 이력서 1차 정리하기”처럼 범위가 좁을수록 잘 작동합니다.
이 장의 결론처음 작업 순서는 완벽한 문서가 아니라, 이 다섯 줄을 채우는 것부터 시작하면 됩니다.
